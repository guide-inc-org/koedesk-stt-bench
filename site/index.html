koedesk STT Bench

Multilingual speech-to-text accuracy, measured under a preregistered methodology — Track A (FLEURS), 13 hosted engine variants × 12 languages × 200 utterances.

Conflict of interest, stated up front: this benchmark is built and funded by koedesk, a dictation product that uses ElevenLabs Scribe v2. We do not pretend to be neutral — instead, everything you need to distrust us and re-run the benchmark yourself is published: the methodology (committed before the headline runs — see the commit history), every raw API response (results/raw), all scoring code, and the terms-of-service audit that decided which engines could be included at all. Where Scribe loses, that result is published unchanged.
Track B is pending. Track A uses public test sets that engines may have trained on; the benchmark's headline question — does the ranking survive on fresh, contamination-proof 2026 audio? — is answered by Track B, which will be published here with the Track A ⇄ Track B divergence reported prominently (§9).

How to read the tables

Engines are ranked per language into tie groups (T1, T2, …): an engine shares a tier when a paired bootstrap (n=10,000, seed frozen at preregistration) cannot statistically distinguish it from the tier leader (95% CI of the paired difference includes zero). Differences inside a tier are not statistically meaningful — including for Scribe v2. The headline metric is CER for Japanese, Chinese and Thai, WER for the other nine languages; both are always shown. Lower is better.

Notes & known artifacts

  • Track A uses FLEURS (public test set), 200 utterances × 12 languages, delivered to every engine as the identical 16-bit PCM file peak-normalized to −6 dBFS (PREREGISTRATION §4). Public test sets may appear in engine training data; Track B (fresh 2026 audio, contamination-proof) is the headline question of this benchmark and is pending.
  • AmiVoice participates in Japanese only (§2). Mistral Voxtral covers 9 of 12 languages — Vietnamese, Indonesian and Thai are not supported by the API and are shown as “not supported”, not as failures (Amendment 5).
  • Thai has no word segmentation; WER is not defined for it (§5.6) and its column shows “—”. CER is the Thai headline.
  • An engine returning empty text scores all-deletions and is never excluded (§6) — excluding empties is the classic way to launder a bad engine.
  • Gemini 2.5 Pro / Chinese includes one utterance (zh_0192) where the engine returned ≈4,800 characters of English reasoning text instead of a transcript (finish_reason=stop, complete response). Published as measured per R-2.4/§7: engine behavior under preregistered default parameters is the thing being measured. Without that single utterance the cell's CER would be 0.0645; with it, 0.4834. This is a real failure mode of LLM-type STT engines.
  • 13/200 FLEURS Chinese references embed parenthetical English glosses that are not spoken in the audio (e.g. “罗兰多·门多萨 (Rolando Mendoza)”). This inflates CER equally for every engine and does not affect ranking.
  • The five Gemini variants are called through Cloudflare AI Gateway (unified billing); their latency figures include the gateway hop. Accuracy is unaffected.
  • RTF = request wall-clock seconds ÷ audio seconds, measured from one machine (Mac mini, Tokyo-region ISP). Informational, not a ranking axis (§6).
  • Prices are vendor list prices per audio-hour at run date. “—” means the vendor has no per-hour list price (token-priced Gemini variants; AmiVoice measured on its free tier).

Reproduce / audit

Everything is in the repository: raw API responses for all 28,400 utterances, frozen normalizers and scoring code with tests, the append-only amendment log, and make_scores.py / make_site.py which regenerate results/scores.json and this page deterministically — no hand-edited numbers. Re-running any cell needs only your own API keys.
